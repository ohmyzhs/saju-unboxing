# 챗봇·리포트·핵심 UX 복구 설계

## 목표

운영에서 실패하는 챗봇 질문 흐름을 복구하고, 질문/답변을 재진입 가능한 턴으로 보존하며, 리포트 생성의 Vercel 타임아웃 구조를 제거한다. 동시에 궁합 프로필 추가, 오늘운세 마음 입력, 홈 튜토리얼을 실제 동작하는 UX로 완성한다.

## 검토한 접근

1. 현재 SSE 한 요청을 유지하며 함수 제한시간만 늘리기: 구현은 작지만 Hobby 제한과 공급자 지연에 계속 취약하므로 제외한다.
2. 모든 생성을 별도 상시 워커로 이전하기: 가장 내구성이 높지만 이번 범위에서 인프라와 배포 체계를 크게 바꾸므로 제외한다.
3. 분석 설계는 빠르게 반환하고 본문을 독립 요청으로 병렬 생성하기: 기존 `/api/saju/section`과 보관함 초안을 재사용하면서 단일 함수의 임계 경로를 줄일 수 있어 채택한다.

## 데이터베이스와 챗봇 실행

- `uuid_generate_v4()`를 PostgreSQL 기본 제공 범위가 넓은 `gen_random_uuid()`로 교체한다.
- 운영 복구용 멱등 migration을 추가한다. 확장, 챗봇 테이블, 주문 fulfillment 컬럼, `site_config.chat_model`, 챗봇 RPC와 권한을 한 번에 보장한다.
- 사용자 메시지와 assistant placeholder는 같은 트랜잭션에서 만들고 `chat_runs`로 한 턴을 연결한다.
- 세션 상세 API는 raw messages뿐 아니라 `{run,user,assistant}` 턴 배열을 반환한다. 재진입 UI는 턴을 기준으로 렌더한다.
- Agent 입력은 선택 리포트 스냅샷, 현재 질문, 이전 완료 턴만 포함한다. 외부 조회는 금지한다.
- 시스템 프롬프트는 한국 사주 명리 상담 전문가 페르소나, 사용자 현실 우선, 보고서와 현실이 충돌할 때의 교정 방식, 이전 맥락 연속성, 안전 규칙을 명시한다.
- 답변 delta는 DB 이벤트로 누적하고 SSE는 `Last-Event-ID`로 재연결한다. UI는 대화방 재진입 시 활성 run을 자동 재구독한다.

## 챗봇 UX

- 챗봇 첫 화면은 잔액/상품/리포트 선택/기존 대화방 목록을 보여주는 로비다.
- 리포트를 선택하거나 기존 대화방을 누르면 별도 풀화면 채팅방으로 전환한다.
- 채팅방은 상단 뒤로가기와 리포트명, 스크롤 가능한 턴 히스토리, 하단 고정 composer로 구성한다.
- 전송 실패 시 질문과 실패 메시지를 화면에 유지하고 재시도 안내를 표시한다.

## 리포트 생성

- `/api/saju/analyze` SSE는 만세력과 짧은 plan까지만 반환하고 종료한다.
- plan 출력 토큰과 제한시간을 실제 산출물 크기에 맞게 줄인다. 공급자가 제한시간 안에 plan을 주지 못하면 제품별 안전한 기본 outline을 사용한다.
- 브라우저는 plan을 즉시 보관함 초안으로 저장하고 `/api/saju/section`을 두 섹션씩 병렬 호출한다.
- 각 section 성공 시 즉시 화면과 보관함을 갱신한다. 일부 실패는 성공한 섹션을 보존하고 재시도 가능한 실패 상태로 남긴다.

## 나머지 UX

- 궁합 화면의 각 프로필 선택 옆에 `새 프로필 추가`를 제공하고, 저장 후 궁합 화면의 해당 슬롯으로 복귀한다.
- 오늘운세 마음 칩은 선택 상태를 표시하고 API에 전달한다. `직접 입력`을 누르면 입력창을 열고 입력값을 캐시 키와 AI 컨텍스트에 포함한다.
- 홈의 초보자 안내는 4단계 튜토리얼 버튼과 설명/실행 CTA로 바꾼다.

## 오류 처리와 검증

- DB 함수 부재와 UUID 함수 부재는 사용자에게 migration 필요 오류 코드로 변환한다.
- 단위 테스트로 migration, 턴 조립, Agent 컨텍스트, plan 제한시간/폴백, UI 계약을 검증한다.
- 전체 테스트, 문법 검사, Vercel FE/API build, 모바일 브라우저 핵심 흐름을 검증한 뒤 main에 병합·푸시한다.
