# 주문·리포트 복구 및 포인트 운영 개선 설계

## 목표

결제 이후 리포트 생성이 실패해도 결제와 생성 상태를 잃지 않고 사용자가 직접 복구할 수 있게 한다. 결제 내역은 주문 상태에 맞는 행동을 제공하며, 무료운세 재생성권과 관리자 포인트 기능은 사용 방법이 명확하고 즉시 반응해야 한다.

## 확인된 원인

- 리포트는 모든 섹션 호출이 끝난 뒤 한 번만 보관된다. 중간 호출이 오래 걸리거나 실패하면 보관함에 아무것도 남지 않는다.
- 결제 완료 직후 임시 구매 정보를 먼저 삭제한다. 리포트 생성 실패 후 다시 시작할 컨텍스트가 없다.
- 결제 내역 카드는 표시 전용 마크업이라 상세, 결제 재개, 취소, 리포트 재시도 기능이 없다.
- 무료운세 재생성 버튼은 결과 맨 아래에 조건부로만 나타나며 토큰 보유 사실과 사용 흐름이 분리되어 있다.
- 관리자 포인트 폼은 비동기 요청 뒤 `event.currentTarget`을 다시 사용해 `null.reset()` 오류가 발생한다.
- 회원을 선택할 때마다 회원과 주문 전체를 다시 조회해 상세 표시가 느리다.
- 관리자 결제 목록은 `points_used`, `pay_method`를 매핑하지 않아 포인트 전액 결제를 0원 주문처럼 표시한다.

## 주문 상태와 사용자 행동

브라우저의 계정별 주문 미러에 결제 상태와 리포트 상태를 분리한다.

- 결제 대기: `결제 진행중`. `결제 이어하기`, `주문 취소` 제공.
- 결제 완료: 현금·포인트·결제수단·주문번호·승인시각 상세 제공.
- 리포트 생성중: 결제는 완료된 상태로 유지하고 `리포트 생성중`을 별도 표시.
- 리포트 완료: `리포트 보기` 제공.
- 리포트 실패: 오류 안내와 `리포트 다시 생성` 제공. 추가 결제를 만들지 않는다.
- 주문 취소: 미승인 주문만 취소할 수 있다. 포인트 전액 결제와 결제 완료 주문은 취소할 수 없다.

`POST /api/orders`는 기존 생성 요청 외에 `action: "cancel"`을 받는다. 로그인 주문은 현재 사용자 소유인지 확인하고, 결제 완료가 아닌 주문만 `결제 취소`로 변경한다. 프론트는 주문 미러에도 같은 상태를 기록한다.

## 리포트 내구성

주문 미러에 상품, 프로필, 상대 프로필, 결제 정보로 구성된 복구 컨텍스트를 저장한다. 리포트 생성 시작 시 `reportStatus: "generating"`을 기록한다.

1. 분석 설계 응답을 받으면 제목·만세력·요약·섹션 자리표시자로 보관함 초안을 즉시 만든다.
2. 섹션 응답이 도착할 때마다 화면과 초안을 갱신한다.
3. 모든 섹션이 끝나면 초안을 완료 상태로 바꾸고 주문을 `reportStatus: "complete"`로 바꾼다.
4. 실패하면 이미 받은 내용은 유지하고 주문을 `reportStatus: "failed"`로 바꾼다.
5. 결제내역의 재시도는 저장된 복구 컨텍스트로 `startAnalysis`만 다시 호출한다. 새 주문이나 새 결제를 만들지 않는다.

네트워크 요청에는 제한 시간을 적용해 무한 대기를 막는다. 배치 섹션 호출이 실패하면 기존 단일 섹션 폴백을 사용하되, 각 호출도 제한 시간 안에 종료한다.

## 결제 내역 UI

주문 카드를 버튼형 요약과 펼쳐지는 상세 영역으로 구성한다.

- 요약: 상품, 대상, 결제 합계, 상태, 시각.
- 상세: 주문번호, 상품금액, 토스 결제액, 사용 포인트, 결제 방식, 승인시각.
- 결제 대기 행동: 결제 이어하기, 주문 취소.
- 결제 완료 행동: 리포트 보기 또는 리포트 다시 생성.
- 포인트 충전 주문: 적립 포인트와 결제 금액만 보여주고 리포트 행동은 제공하지 않는다.

## 무료운세 재생성 UI

무료운세 결과 상단에 재생성권 상태 카드를 둔다. 토큰이 있으면 `재생성권 1개 사용` 버튼과 남은 개수를 표시하고, 누르면 확인 후 캐시를 무시해 새 결과를 생성한다. 토큰이 없으면 충전 화면이나 안내만 표시한다. 포인트 화면에도 `무료운세에서 사용하기` 이동 버튼을 둔다.

## 관리자 포인트 UI와 성능

- 회원 목록은 최초 진입과 명시적 새로고침에만 조회한다.
- 회원 상세 요청은 선택한 회원의 포인트 계정과 거래만 조회한다.
- 선택된 회원을 강조하고 검색 입력, 로딩 상태, 빈 상태를 제공한다.
- 포인트 조정과 토큰 부여 폼은 제출 전에 폼 참조를 보관해 비동기 완료 뒤 안전하게 초기화한다.
- 적용 성공 시 상세만 다시 조회하고 성공 메시지를 표시한다.
- 자주 쓰는 `+1,000`, `+5,000`, `-1,000`, 토큰 `+1` 빠른 버튼을 제공하되 최종 서버 검증은 기존 RPC를 사용한다.

## 관리자 결제 목록

주문 응답에 `pointsUsed`, `payMethod`를 포함한다. 목록은 다음처럼 표시한다.

- 토스: `현금 1,990원`
- 포인트: `1,990pt`
- 혼합: `현금 990원 + 1,000pt`

포인트 전액 주문도 결제 완료 주문으로 집계하되 현금 매출에는 0원으로 반영한다.

## 오류 처리와 보안

- 서버 주문 취소는 로그인 사용자 소유권과 미결제 상태를 모두 검사한다.
- 결제 완료 주문 취소 요청은 409로 거절한다.
- UI 오류 메시지는 사용자 행동과 복구 방법을 함께 표시한다.
- 프로필 복구 컨텍스트는 기존 계정별 `user_data` 범위 안에서만 저장한다.
- 포인트 변경은 기존 원자 RPC와 감사 로그를 유지한다.

## 검증 기준

- 리포트 섹션 일부가 실패해도 보관함 초안과 주문 복구 버튼이 남는다.
- 결제 진행중 주문을 이어서 결제하거나 취소할 수 있다.
- 결제 완료 주문에서 현금·포인트·주문번호와 리포트 상태를 확인할 수 있다.
- 토큰 보유 사용자가 무료운세 화면에서 재생성 방법을 즉시 확인하고 사용할 수 있다.
- 관리자 조정 후 콘솔 오류가 없고 선택 회원 상세만 다시 요청한다.
- 관리자 결제 목록에 포인트 전액·혼합 결제가 구분되어 보인다.
- 기존 포인트 원자성, 환원, 결제 멱등 테스트가 계속 통과한다.
