# 스트리밍 리포트 생성과 결제·무료운세 UI 개선 설계

## 목표

리포트 품질과 분량은 유지하면서 생성 과정을 실제 단계에 맞게 빠르고 투명하게 보여 준다. Vercel 함수가 60초에 강제 종료되는 문제를 제거하고, 완료된 섹션은 즉시 화면과 보관함 초안에 반영한다. 함께 확인된 무료운세 원국 표시, 헤드라인 겹침, 포인트 전액사용 UI도 수정한다.

## 확인된 원인

- `vercel.json`이 모든 API 함수의 `maxDuration`을 60초로 덮어쓰고 있다. Fluid Compute가 표시되는 현재 프로젝트에서는 Hobby도 300초까지 사용할 수 있지만 저장소 설정 때문에 정확히 60초에 504가 발생한다.
- `/api/saju/analyze`는 만세력 계산 뒤 AI 설계 응답 전체를 기다린 다음 JSON을 반환한다. DeepSeek Flash라도 공급자 대기열이나 긴 꼬리 지연이 있으면 첫 응답 전까지 화면에 실제 진행 정보가 없다.
- 설계 요청에 본문 작성용 장문 프롬프트 전체를 재사용한다. 설계 단계는 제목·섹션 각도·개운 근거만 필요하므로 불필요한 입력 토큰과 추론 범위가 크다.
- AI 전송 계층에 제한 시간과 선택적 재시도 기준이 없고 출력 토큰 상한도 호출 목적별로 지정하지 않는다.
- 프론트 진행률은 600ms마다 난수를 더해 92%까지 올린 뒤 네트워크 응답을 기다린다. 따라서 실제 단계와 일치하지 않고 타임아웃 전까지 같은 메시지가 반복된다.
- Vercel의 비 JSON 504 본문을 바로 `response.json()`으로 읽던 운영 `main` 코드 때문에 원래 오류 대신 `Unexpected token`이 노출됐다. 작업 브랜치에는 텍스트 우선 파싱 수정이 있으나 아직 Production Branch에 반영되지 않았다.
- 무료운세 원국의 `summary.pillars`는 `"임신"` 같은 문자열인데 화면 함수는 `{ stem, branch }` 객체만 읽어 모두 `?`를 출력한다.
- 무료운세 점수는 절대 위치이고 긴 헤드라인은 점수 공간을 예약하지 않아 두 텍스트가 겹친다.

## 리포트 생성 구조

브라우저는 `/api/saju/analyze`에 `stream: true`와 `Accept: text/event-stream`을 보내고 하나의 구조화된 SSE 연결을 연다. 원시 모델 토큰은 JSON 경계와 보관 단위가 불안정하므로 그대로 노출하지 않고, 서버가 검증된 업무 이벤트만 보낸다.

이벤트 순서는 다음과 같다.

1. `started`: 요청 검증과 설정 로드가 끝남.
2. `manse_ready`: 한국표준시 기준 만세력 계산이 끝남.
3. `plan_started`: 경량 설계 프롬프트 AI 호출 시작.
4. `heartbeat`: 외부 API를 기다리는 동안 연결 유지와 경과 상태 전달.
5. `plan_ready`: 검증된 헤드라인, 섹션 설계, 만세력, 요약, 컨텍스트 전달.
6. `section_ready`: 1~2개 묶음의 본문 생성과 스키마 검증이 끝날 때마다 전달.
7. `complete`: 모든 섹션 완료.
8. `error`: 단계와 복구 가능한 한국어 메시지를 포함한 실패.

설계가 끝나면 1~2개 단위 섹션 묶음을 병렬 호출한다. 각 묶음 실패 시 해당 묶음만 단일 섹션 호출로 재시도해 이미 성공한 결과를 버리지 않는다. 기존 JSON 응답과 `/api/saju/section` 계약은 호환을 위해 유지한다.

## 실제 진행률과 체크포인트

진행률은 시간이나 난수가 아니라 완료된 업무 단위로 계산한다.

- 요청 시작 5%
- 만세력 완료 20%
- 설계 시작 25%
- 설계 완료 40%
- 각 섹션 완료: 40%에서 95% 사이를 `완료 섹션 수 / 전체 섹션 수`로 배분
- 최종 저장과 완료 100%

`plan_ready`를 받으면 기존 `ReportRecovery` 초안을 즉시 만들고, `section_ready`마다 같은 초안 ID의 본문을 갱신한다. 브라우저를 닫은 뒤에도 서버 작업이 보장되는 백그라운드 잡 구조는 아니므로 기존의 “화면을 나가도 계속 진행” 문구는 제거하고 “완료된 단계마다 보관함에 저장”으로 정확히 안내한다. 실패하면 받은 섹션을 유지하고 결제내역의 리포트 재시도 경로를 사용한다.

## AI 프롬프트와 전송 제한

설계 단계 전용 `buildPlanPrompt()`를 둔다. 상품별 목적, 사람 중심의 쉬운 한국어, 한자·전문용어 금지, 제목 다양성, 바넘·공포 금지, 출력 범위만 포함하고 2,400자 이하를 목표로 한다. 본문 3~4문단 작성 규칙과 상세 예시는 섹션 프롬프트에만 둔다. 관리자 추가 지침은 두 단계 모두 유지한다.

호출별 출력 상한을 지정한다.

- 설계: `maxTokens: 2,048`
- 섹션 묶음: `maxTokens: 4,096`

AI 전송 계층은 호출별 `timeoutMs`를 지원한다. 제한 시간, 429, 5xx만 최대 한 번 재시도하고 인증·입력 오류나 일반 프로그래밍 오류는 재시도하지 않는다. 최종 제한 시간 오류는 504와 한국어 메시지로 변환한다. JSON 파싱·스키마 실패에 대한 기존 1회 재요청은 유지하되 전체 시도 횟수는 무한히 늘어나지 않게 제한한다.

## Vercel 실행 시간

`vercel.json`의 `api/**/*.js` 최대 실행 시간을 300초로 바꾼다. 이는 느린 요청을 정상으로 간주하는 해결책이 아니라 공급자 지연이 60초를 잠깐 넘었을 때 함수가 강제 종료되지 않게 하는 안전망이다. 사용자 체감 속도는 경량 설계 프롬프트, 섹션 병렬 생성, 구조화 SSE로 개선한다.

## 무료운세 UI

원국 표시 앞에 문자열·객체를 모두 받는 정규화 계층을 둔다.

- `"임신"`, `"壬申"`: 천간·지지를 분리해 한자와 한글을 함께 표시
- `{ stem, branch }`: 기존 엔진 객체 형식 유지
- 출생시간 미상인 시주: `?` 반복 대신 `미상`으로 표시
- 예기치 않은 값: 빈 칸이 아니라 `미상`으로 안전하게 표시

히어로는 CSS Grid로 헤드라인과 점수 열을 분리한다. 상단 메타와 태그는 전체 너비를 사용하고, 긴 헤드라인은 점수 열을 침범하지 않고 줄바꿈한다. 작은 화면에서는 점수가 다음 줄로 내려가도록 한다.

## 포인트 전액사용

포인트 입력 옆에 `전액사용` 버튼을 둔다. 누르면 `min(보유 포인트, 상품 총액)`을 적용하므로 포인트가 충분하면 토스 금액이 0원이 되고, 부족하면 남은 금액만 토스로 결제한다. 잔액이 없거나 포인트 기능이 비활성화된 경우 버튼을 비활성화한다. 직접 입력, 전액사용, 상품 변경 모두 기존 `updatePointPayment()` 한 경로로 금액과 결제 버튼을 갱신한다.

## 오류 처리

- SSE 연결 전에 발생한 검증 오류는 기존 JSON 오류로 응답한다.
- SSE 시작 뒤 오류는 `error` 이벤트로 전달하고 스트림을 정상 종료한다.
- 클라이언트 SSE 파서는 청크 경계에 걸친 이벤트와 여러 줄 `data:`를 처리한다.
- 이벤트 스트림이 아닌 응답은 텍스트를 먼저 읽고 JSON이면 메시지를 사용하며, HTML·일반 텍스트 504는 상태 코드 중심의 복구 메시지로 바꾼다.
- 완료된 초안과 주문 결제 상태는 실패 시 삭제하지 않는다.

## 배포 방식

Vercel Git 연동의 기본 설정에서는 `main`이 Production Branch이고 다른 브랜치는 Preview 배포다. 기능 브랜치에서 검증·커밋·푸시한 뒤 원본 체크아웃의 `main`을 fast-forward 병합하고 `origin/main`에 푸시한다. 원본 체크아웃의 사용자 미추적 파일은 건드리지 않는다.

## 검증 기준

- 설계 프롬프트가 장문 본문 프롬프트를 포함하지 않고 정해진 길이와 출력 토큰 상한을 지킨다.
- 제한 시간·429·5xx만 한 번 재시도하며 최종 제한 시간은 504 한국어 오류가 된다.
- SSE 파서가 분할 청크를 읽고 업무 이벤트 순서대로 전달한다.
- 진행률이 `manse_ready`, `plan_ready`, 완료 섹션 수, `complete`에만 따라 변하고 92% 임의 정지가 없다.
- 일부 섹션 실패 시 성공한 섹션과 초안이 유지되고 단일 섹션 폴백이 동작한다.
- 무료운세 문자열 원국이 실제 천간·지지로 표시되고 헤드라인과 점수가 겹치지 않는다.
- 전액사용은 보유액과 상품가 중 작은 값을 적용한다.
- 전체 Node 테스트, 문법 검사, `git diff --check`가 통과한다.
- `main` 푸시 뒤 Production 배포가 시작되고 운영 URL이 정상 응답한다.
